Block Frame Structure
Every block in Field 60 starts with a fixed 7-byte header
Field 60 carries a variable number of parameter blocks. Each block begins with a 7-byte frame header followed by a variable-length data payload. Blocks are parsed sequentially until the hex string is exhausted.

VERSION
Protocol version (e.g. 0001)

Identifies table (01–16)
2 bytes

TABLE DATA
Payload. Offsets inside here start at 0 = tableId byte
ℹ️ The Length field is 4 hex chars (2 bytes) but its value is read as a decimal integer (e.g. 0190 = 190 bytes, NOT 0x0190 = 400 bytes). This is why the parser uses parseInt(lengthStr, 10).
ℹ️ When table parsers use getElement(offset, length), byte offset 0 = the tableId byte itself, bytes 1–2 = the length field, and byte 3 = first actual data field. This is why most field offsets start at 3.

Table 16 — IDS Config
Integrated Data Store operator identifiers
Table ID 0x16
Contains N operator IDS entries. Each entry carries an operator identifier and an IDS value (8 bytes each). The parser loops from entry 0 to count-1, advancing 16 bytes per iteration.
Byte OffsetLengthFieldEncodingNotes
01Table IDHEXAlways 0x16
1–22LengthDECIMAL
31Entry CountDECIMALNumber of IDS entries
↻ REPEAT × COUNT (idx starts at 4, +16 bytes per entry)
idx8Operator IdentifierHEX
idx+88IDSHEX
⚠️
Kotlin source has two bugs in this table:
1. val index = 3 — declared as immutable (val) so it never advances → both Operator ID and IDS always read from the same offset.
2. Loop starts at x = 1, skipping the first entry entirely.
This parser corrects both bugs: uses var idx = 4, loop from 0, advancing +8 per field.
